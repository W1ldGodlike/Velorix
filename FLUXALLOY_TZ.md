## 1. Общая концепция

FluxAlloy — десктопное приложение, **целевая платформенность: Windows (приоритет), macOS и Linux**. Графический комбайн, предоставляющий возможности yt-dlp и ffmpeg без командной строки. Совместность: **Windows 10/11 x64**, **macOS** (Intel и Apple Silicon — отдельные сборки или universal по решению релиза), **Linux x64** (при возможности aarch64 как дополнительный таргет).

Целевой **технологический стек** приложения см. §2. Интерфейс и палитры — §1.1 и §5.

Лицензия проприетарная: бесплатное некоммерческое использование, коммерческое — только с письменного разрешения автора.

**Примечание.** Функции, завязанные на API конкретной ОС (**контекстное меню Проводника**, часть сценариев установщика под Windows и т.д.), перечислены в соответствующих разделах как **«только Windows»**; на macOS/Linux допускаются эквиваленты ниже полного паритета или отложенная реализация (**см. §14**, §19).

### 1.1 UI и UX — консолидация (единый канон)

Ниже — **обязательные принципы** построения интерфейса. Реализация канонически на **Electron + React + TypeScript** (§2); **поведение, иерархия экранов и визуальная модель** не противоречат этому блоку на всех поддерживаемых ОС.

#### Референсы (смысл, не копирование пиксель-в-пиксель)

- **Стиль оболочки:** спокойный «инструментальный» вид в духе **LosslessCut** — тёмный фон как базовое ожидание, мало декоративных элементов, **крупное окно предпросмотра** и **товарищующий ему таймлайн** как центр главного окна.
- **Глубина возможностей:** логическая и по объёму контролов — как у **StaxRip** (кодеки, фильтры, цепочки, пресеты), но **упаковано в свёртываемые секции и аккуратные группы**, визуально **той же палитрой и теми же отступами**, что и превью, без ощущения «две разные программы».

#### Главное окно (обработка / ffmpeg)

| Зона | Назначение |
|------|-------------|
| **Верх** | Полоса действий: источник (**Открыть**), зона статуса «текущий файл или нет источника», при необходимости компактное поле **вставки URL** или явная подсказка «Вставить ссылку — Ctrl+V»; основная кнопка **Старт / Экспорт** (название уточняется локализацией). Дубли второстепенных действий только там, где снижают число шагов. |
| **Центр (доминирует)** | **Видеопредпросмотр** и **таймлайн** — как в LosslessCut: воспроизведение, позиция, обрезка по маркерам (см. §7.2.2); при отсутствии файла — пустое состояние с приглашением открыть/DnD. |
| **Бок или низ основной области** | **Панели настроек кодирования** — вертикальные сворачиваемые блоки («Основные», «Обрезка», «Фильтры», «Аудио», …) или вкладки *внутри* этой панели, но **не** отдельная «вторая программа». Пресеты доступны очевидно (выпадающий список + «сохранить как пресет»). |
| **Вторично** | **Лог/ffmpeg** — узкая подпанель или отделяемая вкладка «Вывод», **моноширинный** шрифт; не перекрывает превью в обычной работе. |
| **Вниз** | **Строка состояния** (§4.C) без дублирования всей инфологии сверху. |

**Приоритет сценария:** один активный файл и один «проект обработки» на экране. **Табличный список из нескольких файлов и пакет** — включаемый режим (§7.3), а не главный первый экран.

#### Вкладка менеджера загрузок (yt‑dlp)

- Основной сценарий — вкладка **«Загрузки»** в верхней навигации единого workspace рядом с **«Редактор»**; отдельное окно из меню/pop-out допускается для мониторинга, но не является главным UX.
- Верх:** многострочное поле URL** + блок быстрых действий («в очередь», старт очереди). Средина:** таблица очереди** (статусы, прогресс). Низ или бок:** сворачиваемые блоки опций yt‑dlp**; эксперт‑строка и предпросмотр команды (§6.3–6.4). Мини‑предпросмотр по возможности не спорит с главным окном по стилю.
- Связки «скачать и открыть в обработчике» и «скачать и сразу обработать» — **здесь** (§6.4), не в главной тулбаре.

#### Прогрессивное раскрытие

- По умолчанию показывать **режим без лишних полей** (или запомненный «расширенный» профиль пользователя из настроек). Режим «все параметры ffmpeg» доступен через **точечные секции**, а не простынёй на одном экране без скролла.
- При первом входе — краткая подпись‑подсказка у ключевых контролов; везде **тултипы** на русском (и EN при EN‑локали).

#### Темизация и визуальная стабильность

- Строго **две темы:** **Тёмная** и **Светлая**. Набор семантических **токенов** один: `Background`, `Surface`, **`SurfaceElevated`** (или `SurfaceAlt` для вложенных панелей), `Border`, **`BorderSubtle`** (ещё менее заметная граница), `TextPrimary`, `TextSecondary`, при необходимости **`TextMuted`**, **`Accent`**, `Danger`, `Success`, состояния `Hover`/`Focus`/`Disabled`. Третья («тёплая») тема из старых редакций ТЗ **не требуется**; акцент тёплоты можно вынести в опцию «слегка сепия фон светлой темы» позже только при отдельном решении продукта.
- **Характер палитры (LosslessCut + редакторский спокойствие типа Cursor):** **лаконично**, малая декоративная насыщенность; соседние слои фона (**`Background`** / **`Surface`** / **`SurfaceElevated`**) отличаются **небольшим** шагом по яркости, без **резких перепадов** и без **«вырвиглазных»** второстепенных цветов или неона. Не использовать чистые **#000000** на полях приложения как основной фон (кроме осознанно отделяемых зон, например содержимого видеопотока). В светлой теме избегать «стерильной» голой белизны везде — допустим очень слегка тёплый или серо‑нейтральный офф‑вайт для `Background`.
- **`Accent`** — единственный яркий «ключ» интерфейса применяется **точечно**: основное действие, активное состояние, акцентные ссылки там, где это уместно. **Не** заливать крупные панели несколькими разными акцентами подряд.
- **Иерархия текста:** **`TextPrimary`** — основное чтение; **`TextSecondary`** — подписи к полям, подсказки, вспомогательная строка статуса (читается, но **не отвлекает**); **`TextMuted`** при необходимости — третья ступень (мелкие подписи, время в статусе и т.п.). Выбор **яркости** текста между уровнями — плавный, без эффекта «в лоб три разных цвета параграфов» на одном экране (кроме осознанного выделения предупреждения/ссылки).
- **Иконографика:** единый набор, один вес линии; по умолчанию иконки ближе по тону к **`TextSecondary`**, активное состояние — **`TextPrimary`** или **`Accent`**, без многокрасочных набора там, где задача техническая и спокойная.
- **Типографика:** один основной **UI‑шрифт**, **моноширинный только** для путей, CLI, логов. Конкретная гарнитура не фиксируется (Segoe / Inter‑класс / системная).

#### Геометрия: прямые линии и скругления

- **Где выпрямлять (как в хорошем редакторском UI типа Cursor):** границы **рамки превью**, **таймлайн** как основная техническая шкала, **меню строка**, вертикальные **разделители между крупными зонами** (превью / панели настроек), **ячейки таблицы** в менеджере загрузок — визуально **строгая сетка** и **перпендикуляры** там, где важно «чертёжность» инструмента.
- **Где смягчать:** **кнопки**, поля ввода, чипы, небольшие группирующие карты настроек — лёгкое **скругление** по правилам следующего блока. Итог: **ни «всё таблеткой»**, ни старый плоско‑угловатый офис‑стиль.

#### Детали оформления (кнопки, скругления, иконки)

Речь о **характере** интерфейса (**LosslessCut** по мягкости контролов + **редко‑акцентный** спокойный стиль палитры, как воспринимается в **Cursor**), без копирования чужих ассетов.

- **Скругления:** у **кнопок и полей ввода** — умеренный радиус **6–10 px** (в логических пикселях до DPI); у **больших панелей** (область вокруг превью, сворачиваемые блоки настроек, группы в окне загрузок) — **slightly larger** (**8–12 px**) или мягкая прямоугольность с лёгким скруглением, **без** «таблеток iOS». Обрамление окна (title bar / traffic lights на macOS) — **в духе нативных ожиданий платформы**, но внутренний контент сохраняет единые токены §5.
- **Кнопки — иерархия:**
  - **Основное действие** (Экспорт / Старт / ключевое подтверждение): заливка **`Accent`** (контрастный текст белый или тёмный в зависимости от темы); **без глянцевых градиентов**.
  - **Вторичные действия**: «призрак» (**прозрачный** фон, тонкая **граница** `Border` или лёгкий фон **`SurfaceElevated`**), при наведении чуть заметнее фон или граница.
  - **Третичные / только иконка** (перемотка, мелкий служебный контроль): минимальный padding, hover — круг или скруглённый квадрат с **`SurfaceElevated`**.
  - **Опасное действие:** отдельный токен `Danger`; не красить всё приложение красным, использовать только там, где смысл «удалить / прервать без повтора».
- **Иконки у кнопок:** там, где уже есть понятная пиктограмма (Открыть, Настройки, Плей, Ножницы для обрезки и т.д.) — **иконка + короткая подпись** или только иконка с **обязательным тултипом**. Выравнивание: иконка **слева** от текста в горизонтальных кнопках; одинаковый **оптический размер** иконки **≈16–20 px**.
- **Поля и выпадающие списки:** тот же радиус, что у кнопок; при фокусе — **ясное кольцо** или контрастная граница (`Focus`), не только смена цвета текста.
- **Разделители:** тонкая линия **`Border`** на 1 px или два соседних оттенка **`Surface`** вместо тяжёлых цветных полос между зонами (тулбар / превью / панель настроек).
- **Плейсхолдер пустого превью:** нейтральный **`SurfaceElevated`** и короткая подсказка, без кричащих иллюстраций.

Отдельным правилом: **единые** значения радиуса и высоты строки для кнопки в главном окне и в окне менеджера загрузок — чтобы второе окно ощущалось **той же программой**, а не «другим скином».

#### Ошибки и обратная связь

- Сообщения об ошибке: **одна понятная строка** пользователю + кнопки «Копировать детали» / «Открыть лог» где уместно. Долгие операции показывают **фазу** (демукс, кодирование…) и ETA по возможности, без ложной точности.

#### Доступность и масштаб

- Интерфейс читается при **системном масштабе** каждой ОС (ориентир **100–200% эквивалента** там, где ОС задаёт DPI); активные элементы имеют явный focus. Уважать по возможности **уменьшение анимаций** системы / пользователя — при длинном кодировании избыток анимаций в UI отключать (смысл §4.4).

## 2. Среда, инструменты и проект

### 2.1 Целевые платформы и матрица сборок

- **Приоритет разработки:** **Windows** 10 / 11, **64-bit**.
- **Дополнительные цели:** **macOS** (отдельные или universal-сборки для Intel / Apple Silicon), **Linux** (минимум **x64**; **aarch64** — по возможности пайплайна).
- Общее приложение должно использовать **общий код** интерфейса и логики; различия — через слой **`platform` / `nativeMain`** (IPC, пути к `ffmpeg`/`yt-dlp`, открыть в Finder/Files explorer, упаковка).
- Коммерческое распространение на **macOS** извне App Store потребует **подписи и нотаризации** Apple (вне содержательного объёма ТЗ, но заложить в CI дорожную карту).

### 2.2 Технологический стек (канон)

- **Оболочка:** **Electron** (версию зафиксировать в `package.json` / политике безопасности обновлений проекта — LTS или текущая стабильная ветка, согласованная командой).
- **Интерфейс пользователя:** **TypeScript**, **React** — одна кодовая база для **Windows, macOS, Linux**.
- **Архитектура:**
  - **Main process:** Node.js/Electron — запуск процессов `ffmpeg`, `yt-dlp`, `ffprobe`, работа с файловой системой, окна, автобновление (если включено), доступ к платформенным диалогам. **Строгое ограничение:** renderer не получает прямого `require('child_process')`; только **whitelist IPC**.
  - **Renderer:** React-приложение — экраны, состояние, визуальный канон §1.1.
  - **Preload скрипт** — безопасный мост событий `contextBridge`.
- Логическая структура кода напоминает **MVVM** в смысле разделения: **модели / представление / состояние (store или hooks) / сервисы** под каждый крупный инструмент (**yt-dlp, ffmpeg/ffprobe**, пресеты, терминал, инспектор). Конкретная библиотека состояния (**Zustand, Redux Toolkit, Jotai** и т.д.) — на усмотрение реализации при сохранении тестируемости.
- Локализация: ресурсы **JSON** или **Fluent**‑подобные каталоги (`locales/ru/*.json`, `locales/en/*.json`); русский по умолчанию. **Не использовать** `.resx` как обязательный формат. Смена языка без перезапуска (§4.C).
- **Git** в корне проекта; папка **`.cursor/rules/`** актуализировать под Electron/React/TS.
- Комментарии к публичным модулям и ключевым фрагментам main-процесса — на **русском** (или согласованный язык проекта); при генерации огромных файлов — возможность продолжения по частям.

### 2.3 Устаревший стек (не использовать для нового UI)

Исторический вариант **.NET 8 + WinUI 3 + MVVM Toolkit + `.resx`** в данном каноне **не является целевым** для графической оболочки; отдельные утилиты на C# могут существовать как вспомогательные только если это явно зафиксировано скриптом сборки (**не как правило из ТЗ**).

## 3. Управление зависимостями (КРИТИЧНО)

- Релизная сборка по умолчанию поставляется с проверенными `ffmpeg.exe`, `ffprobe.exe` и `yt-dlp.exe` в подпапке `bin` внутри папки программы (`resources/bin`). Используются только относительные пути.
- При первом запуске программа сначала проверяет ручные пути из настроек, затем bundled `bin`, затем пользовательский `userData/bin`. Если движки отсутствуют или пользователь хочет обновить их отдельно от приложения, доступна кнопка скачивания/обновления (yt-dlp — GitHub Releases; ffmpeg — список источников/зеркал, например GitHub build mirror + gyan.dev fallback) с индикацией прогресса. После загрузки выполняет `--version` и выводит «Готово».
- После загрузки/подготовки релизных бинарников проверяется SHA256-хеш файлов или архивов. Доверенные хеши получить со страниц релизов на момент подготовки сборки и зафиксировать во внешнем JSON. Предусмотреть механизм обновления доверенных хешей без перекомпиляции (например, через JSON-файл, обновляемый при проверке обновлений).
- В настройках: кнопка «Проверить обновления» (сравнивает версии и обновляет движки в фоне), возможность удалить/заменить скачанные в `userData/bin` движки и вручную указать альтернативные пути к движкам с проверкой их работоспособности.

## 4. Главное окно и глобальные элементы

### 4.A Разделение ролей окон

- **Главное окно — обработка (ffmpeg):** визуальный ориентир **LosslessCut** (полноценное **окно предпросмотра** с видеоплеером и **таймлайном**) плюс **глубина настроек** в духе **StaxRip** через компактные сворачиваемые секции в **едином стилевом языке** с предпросмотром (не «разные приложения», а один продукт).
- **Вкладка yt‑dlp** (менеджер загрузок) — равноправная рабочая область единого интерфейса рядом с вкладкой обработки. Отдельное окно из меню/кнопки **pop-out** остаётся дополнительным режимом для мониторинга. В рабочей области загрузок — многострочный ввод URL, **очередь**, настройки скачивания, свой прогресс/лог и при необходимости мини‑предпросмотр (см. §6).

### 4.B Единая зона источника данных в главном окне

- Пользователь должен суметь начать работу любым из способов: **«Открыть файл»**, **Drag‑and‑Drop** файла (и при поддержке контекста — папки) в главную область **или вставки ссылки** (Ctrl+V) в выделенное поле / зону приёма.
- Если выбран **локальный файл** — открывается в **окне предпросмотра**, далее применима вся обработка (§7).
- Если передан **URL** потокового видео:
  - **По умолчанию:** активируется вкладка **«Загрузки»** с этой ссылкой уже добавленной (в поле очереди или первой строкой списка).
  - **Опционально в настройках:** «Одиночная загрузка в рабочую папку и открыть в обработке» — фоново скачать один ролик до локального файла, затем автоматически подставить его как **текущий файл** главного окна (без обязательного открытия менеджера).
- Связка «получено из сети → сразу крутить в ffmpeg» сохраняется как **режим менеджера загрузок** (см. §6.4 «Скачать и сразу обработать»), чтобы не смешивать сложность очередей с главным интерфейсом.

### 4.C Прочее
- Стартовый размер главного окна 1920×1080 (FHD), адаптивная вёрстка. Полная поддержка масштабирования 100%/125%/150%/200% — ни один текст и ни одна кнопка не обрезается.
- **Верхнее меню** (строгая логическая группировка, все пункты с иконками):
  - «Файл»: **Менеджер загрузок (yt‑dlp)…** (или эквивалентное название), Выход — при наличии смысла также «Открыть…» здесь же.
  - «Инструменты»: Анализировать (ffprobe), Терминал, Очистка кэша.
  - «Сервис»: Планировщик задач, Конструктор сценариев, Импорт/Экспорт конфигурации.
  - «Справка»: База знаний, История задач, О программе.
- **Строка состояния** внизу:
  - Версии движков.
  - Индикатор активности и текущий язык.
  - Индикатор текущего кодировщика: «Обработка: CPU (программный)» или «Обработка: NVIDIA NVENC» и т.д. При наведении на индикатор — тултип с моделью GPU (например, «NVIDIA GeForce RTX 3060»), версией драйвера и лимитом одновременных сессий кодирования (если применимо). При автовыборе рядом с названием кодера отображается зелёная метка `[АВТО]`.
- **Языки**: русский и английский. Переключатель в настройках, смена без перезапуска программы. Реализовать через **локализацию, описанную в §2** (JSON / i18next или эквивалент). Все строки интерфейса — в каталогах локализации, не в литералах JSX.
- **Дизайн и стиль** (детально §1.1):
  - Все интерактивные элементы с **логичными иконками** одного набора на весь продукт.
  - **Единые** отступы (сетка 8 px как ориентир), скругления панелей, размер базового текста **14–16 px** до масштабирования DPI, все цвета из **активной темы** §5. Никаких разрозненных стилей между главным и окном загрузок.

### 4.1 Запоминание настроек при любом закрытии

Все состояния интерфейса (галочки, списки, поля, размеры окон, выбранные папки, состояние раскрытых/свёрнутых панелей, активные задачи с текущим статусом) сохраняются при любом закрытии, включая нажатие на крестик. При следующем запуске восстанавливаются. Хранить в **`userData`** Electron (или эквивалент каталога приложения на каждой ОС), формат предпочтительно **JSON** (например `session.json`, `settings.json`).

### 4.2 Подтверждение закрытия

При нажатии на крестик или Alt+F4 — диалог «Закрыть FluxAlloy? Есть активные задачи.» (если есть) или «Закрыть FluxAlloy?» (если нет). Кнопки «Да»/«Нет». Отключение подтверждения — в настройках.

### 4.3 Mini Player

При длительных задачах — возможность свернуть программу в компактную плашку поверх всех окон с общим прогрессом и скоростью. Включение/выключение режима «Поверх всех окон» — через контекстное меню (ПКМ) на плашке.

### 4.4 Производительность интерфейса

- Для редко открываемых тяжёлых панелей использовать **ленивое монтирование** React-компонентов (динамический `import`, условный рендер).

- Во время длительных фоновых операций автоматически ослаблять или отключать **дорогую анимацию** в UI (CSS/React transitions).
- Использовать градиентную заливку прогресс-бара во время активных задач.

### 4.5 Раздел «О программе» и версионирование

- В меню «Справка» → «О программе» отображается:
  - Название и версия в формате `Версия 1.0.0 (сборка 20260508.001)`.
  - `1.0.0` — публичная семантическая версия MAJOR.MINOR.PATCH.
  - `20260508.001` — внутренний идентификатор сборки (дата и порядковый номер за день).
- Создать механизм хранения текущей версии (например, в файле `version.txt` или константе) и автоинкремента номера сборки при каждом билде. Сброс счётчика в полночь реализуется через сохранение даты последнего сброса.
- Там же — кнопка «Экспорт логов для поддержки» (см. п. 18).

### 4.6 Настройки программы (окно-шестерёнка, открывается из меню «Файл» или строки состояния)

- Раздел «Общие»: язык интерфейса, тема (**тёмная / светлая**), звуковой фидбек, подтверждение закрытия, порог нагрузки для рекомендации кодера (по умолчанию 80%), пункт **поведение при вставке URL** (открыть менеджер загрузок против «скачать одним файлом в обработку» — см. §4.B).
- Раздел «По умолчанию»: качество видео, формат контейнера, чекбоксы «Всегда проверять целостность», «Всегда использовать имперсонацию», путь сохранения по умолчанию.
- Раздел «Зависимости»: пути к движкам, кнопка «Проверить обновления», кнопка «Обновить конфигурацию GPU».
- Раздел «Горячие клавиши»: переназначение комбинаций.

## 5. Темизация (ДВЕ ТЕМЫ)

Стратегический канон см. §1.1 («Темизация и визуальная стабильность», палитра, геометрия, иерархия текста).

- **Тёмная тема** (ориентир по палитре; тон задают **LosslessCut** и **спокойные тёмные темы редакторов**, не «яркий маркетинг»): фон **`Background` ~ #1E1E1E** (не ~~#000~~); **`Surface`** / **`SurfaceElevated`** — ступени **#252526 / #2D2D30** или близкие; **`TextPrimary` ~ #E6E6E6–#F0F0F0**, **`TextSecondary`** — приглушённый серый **~ #9D9DA2–#BBBBBB**, **`Accent`** один сдержанный (системный или **~ #0078D4**‑класс), границы **`Border`** **~ #3F3F46**, для едва заметных линий — **`BorderSubtle`** темнее фона лишь на шаг (уточняется таблицей токенов в реализации).
- **Светлая тема**: **`Background`** **~ #FAFAFA–#FFFFFF** (допускается лёгкий нейтральный офф‑вайт по §1.1); **`Surface`** **#F3F3F3**–**F5**; **`TextPrimary`** **~ #1A1A1A**; **`TextSecondary`** **~ #5C5C5C–#737373**; границы **#E0E0E0** / более светлые для `BorderSubtle`; **`Accent`** — тот же по смыслу, что в тёмной, для узнаваемости приложения в обеих темах.
- Переключение в **«Настройки»**, мгновенно, без перезапуска. **По умолчанию для нового пользователя** — тёмная тема (соответствие ожиданиям интерфейса в духе LosslessCut).

> Ранее упомянутая «тёплая» (Eye Comfort) тема исключена из обязательного объёма; при необходимости возвращается отдельной задачей продукта.

## 6. Окно менеджера загрузок (yt‑dlp)

Окно открывается из меню главного приложения (см. §4.A). Отделяет сценарии **массового скачивания**, **очереди** и экспертных ключей yt‑dlp от основного пайплайна обработки с предпросмотром.

*Все элементы управления в этом окне должны иметь понятные русские тултипы, объясняющие их назначение, и логичные иконки.*

### 6.1 Основная панель

- Многострочное поле URL (Drag‑and‑Drop, Ctrl+V). Подпись: «Вставьте ссылки на видео (по одной на строку)».
- При активации окна (получении фокуса), если в буфере обмена есть URL, он может автоматически предлагаться ко вставке в поле ввода — без вторжения с буфера в главное окно обработки, если там работают с локальным файлом.
- **Галочка «Имперсонация браузера»** под полем URL (использует `--impersonate chrome` в yt-dlp, полностью маскирует программу под настоящий Chrome на сетевом уровне).
- Кнопка «Добавить в очередь» → заполняется таблица очереди (колонки: №, Ссылка, Статус, Качество, Прогресс, Действия).
- Кнопки управления очередью над таблицей: «Удалить выбранное», «Очистить всё», «Переместить вверх/вниз».
- **Предпросмотр**: мини‑плеер на **элементах воспроизведения, поддерживаемых Chromium** (`<video>` / MSE по возможности), по прямой ссылке после `yt-dlp -g`, без полного скачивания. Расположен справа от списка или в раскрывающейся панели.
- В колонке «Действия» для каждой задачи — иконки (▶️ Старт, ⏸️ Пауза, ❌ Удалить) с тултипами. При нажатии «Старт» на незавершённой задаче автоматически используется докачка (`--continue`). При нажатии «Пауза» задача приостанавливается с возможностью продолжения. При нажатии «Удалить» предлагается также удалить недокачанный файл.

### 6.2 Настройки скачивания

Все настройки сгруппированы в три сворачиваемые секции:

- **Основные** (то, что меняют чаще всего):
  - Качество видео (выпадающий список): Лучшее (best), 4K (2160p), Full HD (1080p), HD (720p), 480p, Только аудио (best audio). Каждый пункт с русским тултипом.
  - Формат контейнера: Исходный, MP4, MKV, WEBM, AVI, MOV. Каждый с тултипом, описывающим `--merge-output-format`.
  - Чекбокс «Скачать субтитры» (`--write-subs`), при включении — поле выбора языка.
  - Чекбокс «Встроить субтитры» (`--embed-subs`), активно только при включённом «Скачать субтитры».
  - Чекбокс «Извлечь аудиодорожку» (`-x`), при включении — выбор формата (mp3, aac, opus, flac, wav).
  - Чекбокс «Скачать фрагмент» (`--download-sections`): при включении появляются поля для ввода начала и конца фрагмента (например, `*1:30-2:00` или `*5:00-end`). Поля снабжены тултипами с примерами формата.
- **Сеть и прокси**:
  - Чекбокс «Ограничить скорость» (`--limit-rate`), при включении — поле ввода (например, 5M).
  - Прокси: поле для адреса и флаг `--proxy`.
  - Чекбокс «Использовать cookies.txt» (`--cookies`), кнопка выбора файла.
- **Дополнительно**:
  - Чекбокс «Скачать превью» (`--write-thumbnail`).
  - Чекбокс «Пропускать прямые трансляции» (`--no-live`).
  - Чекбокс «Видео не старше даты» (`--dateafter`), DatePicker.
  - Чекбокс «Сухой прогон (dry run)» (`--skip-download`).
  - Чекбокс «Проверять целостность после загрузки».
  - Поле «Макс. число загрузок из плейлиста» (`--playlist-end`).
  - Поле «Шаблон имени выходного файла» с примерами переменных: `%(title)s`, `%(id)s`.

### 6.3 Экспертный режим (раскрывающаяся панель)

- Текстовое поле для любых дополнительных ключей yt-dlp.
- Ссылка на официальную документацию.
- Предпросмотр собираемой команды в реальном времени.

### 6.4 Прогресс, лог и комбинированный режим

- Прогресс-бар текущего файла и общий прогресс-бар (суммирует прогресс всех активных задач).
- Метки: скорость, ETA, размер.
- Расширяемый лог stdout/stderr. Кнопка «Прервать всё».
- Чекбокс «Скачать и сразу обработать» — открывает панель с базовыми настройками ffmpeg (пресет, обрезка, сжатие). Команды выполняются последовательно.

## 7. Главное окно: обработка (ffmpeg)

Размещается в **главном окне** приложения рядом с **окном предпросмотра и таймлайном** (§4).

*Все элементы управления в этой области должны иметь понятные русские тултипы, объясняющие их назначение, и логичные иконки.*

### 7.1 Основная панель

- **Приоритет сценария:** работа с **одним текущим файлом/проектом** (выбор в §4.B), предпросмотр и параметры экспорта. **Пакетная таблица** и очередь на несколько файлов — **дополнительный режим** (см. §7.3), не главный фокус первого экрана.
- Кнопки «Добавить файлы» и «Добавить папку» (поддержка Drag‑and‑Drop файлов и папок прямо в таблицу в пакетном режиме; в простом режиме — то же действие трактовать как замену **текущего** источника).
- Таблица исходных файлов (колонки: Файл, Размер, Длительность, Кодек, Статус, Действия). Данные о кодеке — из медиа-инспектора.
- Кнопки управления: «Удалить», «Очистить», «Переместить вверх/вниз».
- В колонке «Действия» для каждой задачи — иконки (▶️ Старт, ⏸️ Пауза, ❌ Удалить) с тултипами. При нажатии «Пауза» процесс приостанавливается. При нажатии «Удалить» предлагается удалить временные файлы.

### 7.2 Панель настроек (пресеты и вкладки)

- Пресеты (выпадающий список): Без пресета, TikTok, YouTube, YouTube Shorts, ВК Видео, ВК История, Instagram Reels, Telegram канал, Сжатие для Telegram, Сжатие для Discord, Экспорт в iPhone, Архивное сжатие. Кнопка «Сохранить текущие настройки как пресет». Каждый пресет с тултипом.
- При создании системных пресетов использовать актуальные официальные рекомендации платформ по кодекам, битрейтам, разрешениям и форматам. Если точные цифры неоднозначны, привести оптимальные значения с комментарием.
- **Кнопка «🔍 Анализировать»** (интегрирована в панель пресетов): при нажатии запускает `ffprobe` для выбранного файла, анализирует его характеристики (чересстрочность, битрейт, разрешение) и показывает рекомендацию в сплывающем окне (например, «Рекомендуем пресет Деинтерлейсинг YADIF»). Пользователь может принять или отклонить.
- Логика взаимодействия с автовыбором кодека (см. 7.2.1): при выборе пресета платформы чекбокс «Автовыбор оптимального кодека» автоматически включается и блокирует ручные настройки. Если пользователь вручную снимает эту галочку, выводится диалог-предупреждение: «При ручном выборе кодека пресет больше не гарантирует оптимальную совместимость с платформой. Продолжить?» с кнопками «Да»/«Нет» и галочкой «Больше не показывать до перезапуска программы». При «Нет» автовыбор остаётся включённым.
- Внутренние вкладки (все с русскими тултипами и примерами для сложных опций). Для бинарных настроек использовать маленький **pill switch** (`вкл/выкл`, семантика checkbox/switch), а не `select` из двух вариантов; выпадающие списки оставлять только там, где реально больше двух вариантов.

#### 7.2.1 Основные

- Видеокодек (список: H.264, H.265, AV1, VP9, Copy и др.), с тултипами, какой для чего лучше.
- Аудиокодек (AAC, MP3, Opus, FLAC, Copy и др.), с тултипами.
- Контейнер (MP4, MKV, MOV, AVI, WebM, GIF).
- Битрейт видео (поле ввода + ползунок).
- CRF (ползунок 0–51, метки «Лучше качество» / «Меньше размер»).
- Папка сохранения, чекбокс «Сохранять в исходную папку».
- Шаблон имени выходного файла (переменные: `{original_name}`, `{date}`, `{playlist}`, `{uploader}` и т.д.).
- Pill switch «Автовыбор оптимального кодека» (при включении список кодеков неактивен, программа выбирает сама; логика блокировки пресетом — см. выше).
- Кнопка «Оценить»: запускает бенчмарк на коротком семпле из исходного видео (первые 30 секунд или случайный отрезок). Для каждого доступного в системе кодера (CPU, NVENC, AMF, QSV) выполняется тестовое кодирование. Замеряется скорость (кадров/сек), вычисляется общее время для всего файла, загрузка CPU/GPU (пиковая и средняя), примерный размер выходного файла. Результаты выводятся в таблице/списке. Оптимальный вариант (наименьшее время при условии, что нагрузка не превышает заданный в настройках порог) выделяется зелёной иконкой и подписью «Рекомендовано». При клике на вариант соответствующий кодер автоматически выбирается. Если доступен только CPU, выводится только его оценка с пометкой «Единственный доступный».

#### 7.2.2 Обрезка и склейка

- Визуальный таймлайн с двумя маркерами (начало/конец) и покадровой навигацией в мини-плеере (стрелки влево/вправо — на 1 кадр).
- Поля `-ss` и `-to`. Кнопка «Добавить фрагмент» (для склейки нескольких).

#### 7.2.3 Видеофильтры

- Масштабирование (ширина × высота, сохранять пропорции).
- Визуальный конструктор кропа (нарисовать область на превью и применить).
- Поворот (0–360°).

- Изменение FPS.
- Цветокоррекция: яркость, контраст, насыщенность (ползунки).
- Деинтерлейсинг: Нет, YADIF, BWDIF (с тултипами).
- Стабилизация: кнопки «Анализировать видео» и «Применить стабилизацию».
- Шумоподавление (hqdn3d, nlmeans).
- Возможность изменять порядок фильтров перетаскиванием (drag-and-drop).

#### 7.2.4 Аудиофильтры

- Нормализация (EBU R128, RMS).
- Изменение скорости/темпа с сохранением/изменением тональности.

#### 7.2.5 Субтитры и метаданные

- Конвертация субтитров (SRT ↔ ASS).

#### 7.2.6 Цвет и HDR

### 7.3 Пакетная обработка

- Все настройки применяются ко всем файлам.
- Параллельная обработка: очередь задач **producer–consumer** с настраиваемым **максимумом одновременных заданий** (1, 2, 4, авто). Каждый файл — отдельная асинхронная задача (реализация на **Node/worker threads** или ограничитель в main‑процессе — на усмотрение, без блокировки renderer).
- Режим экономии ресурсов: опция в настройках, которая форсирует `-threads 1` для ffmpeg и снижает приоритет процессов.
- Умная очередь: перетаскивание файлов мышью для изменения порядка.
- При возникновении ошибок в пакетном режиме используется накопительная панель ошибок: в конце обработки (или при остановке) выводится сводка «Завершено с ошибками (X из Y)». Можно выбрать любую ошибку из списка и нажать кнопку поиска в Google/DeepSeek. Для одиночных задач — диалог сразу после сбоя (см. п. 18).
- При отмене активной задачи или возникновении критической ошибки автоматически удаляются временные/недокачанные файлы.

### 7.4 Комбинированный режим

- Чекбокс «Предварительно скачать» + поле URL. Сначала yt-dlp, затем ffmpeg.

### 7.5 Изображения (полный спектр)

- Поворот, отражение, настройка качества/сжатия, конвертация между форматами.
- Создание слайд-шоу из изображений (настраиваемая длительность, эффекты переходов).
- Генерация превью-спрайтов из видео (сетка кадров с временными метками).
- Извлечение кадров из видео (см. 7.6).

### 7.6 Извлечение кадров (Snapshot)

## 16. Аппаратное ускорение (авто/ручное) и оптимизация

- В настройках раздел «Аппаратное ускорение».
- При первом запуске и по кнопке «Обновить конфигурацию» выполняется микродиагностика: FluxAlloy генерирует крошечный тестовый файл в несколько кадров и пробует закодировать его через `h264_nvenc`, `hevc_amf`, `h264_qsv` (если соответствующие кодеки найдены в `ffmpeg -encoders`). Замеряется время и факт успеха. Проверяются профили 4:2:0 и 4:4:4, 8/10 бит, HDR. Результат — матрица доступных кодеков и их лимитов.
- **Авторежим**: программа сама выбирает лучший доступный кодер:
  - Приоритет: дискретный GPU (NVENC > AMF), затем интегрированный (QSV), затем CPU.
  - Учитывает ограничения: NVIDIA от GTX 1050 и драйвер 450.80+, AMD от RX 6000 с AMF 1.4.24+, Intel QSV от Sandy Bridge.
  - Если GPU нет — только CPU.
- **Ручной режим**: пользователю показывается полный список кодеков (CPU, NVENC, AMF, QSV — только те, что успешно прошли диагностику). Для каждого — подробнейший тултип (см. ниже). В ручном режиме доступен чекбокс «Режим экономии ресурсов», форсирующий `-threads 1` и снижающий приоритет процессов.
- **Тултипы для каждого кодера** (должны быть в интерфейсе и в Базе знаний):
  - **CPU (программное кодирование)**: Самый совместимый и гибкий метод. Эталонное качество (libx264, libx265). Медленная скорость (1×). Высокая нагрузка на CPU. Использовать для финального экспорта, мастер-копий, сложных фильтров (стабилизация, шумодав).
  - **NVIDIA NVENC**: Аппаратный кодер на картах NVIDIA (GeForce GTX 1050+). Лучшее среди GPU-кодеров качество, особенно для HEVC. До 10× быстрее CPU. Ограничение: 2-3 одновременные сессии на потребительских картах. Использовать для быстрых задач, пережатия множества файлов, HEVC и AV1 на RTX 40xx.
  - **AMD AMF**: Аппаратный кодер на картах AMD (RX 6000+). Приемлемое качество для H.264 (уступает NVENC и QSV). Для HEVC — хорошее, но возможны сбои. Высокая скорость. Ограничения: качество H.264 «хромает» на старых архитектурах, при низких битрейтах — артефакты. Использовать если у вас AMD-карта и нет альтернатив. Для HEVC — смелее, для H.264 — когда скорость важнее качества.
  - **Intel QSV**: Аппаратный кодер, встроенный в процессоры Intel (Sandy Bridge+). Стабильно хорошее качество для H.264, сопоставимое с NVENC. Для HEVC — компромисс. До 7× быстрее CPU. Ограничения: качество HEVC хуже, чем у NVENC. Зависит от поколения процессора (до 6-го — без HEVC). Использовать на ноутбуках с Intel без дискретной карты, для повседневной конвертации, записи стримов.
- **Сравнительная оценка**: при нажатии кнопки «Оценить» в блоке основных параметров кодирования (п. 7.2.1) дополнительно показывается таблица сравнения времени и нагрузки для каждого доступного кодера с выделением рекомендуемого.
- Порог нагрузки для выделения «Рекомендовано» (по умолчанию 80%) настраивается в разделе Настроек «Общие».
- В строке состояния всегда отображается текущий активный кодер с меткой `[АВТО]` при автовыборе.

## 18. Логирование и диагностика

- Формат в файле: структурированный **JSONLines** / JSON (одна строка лога — одно событие). Рекомендуемые движки: **`electron-log`**, **pino** или **winston** в **main process** (и при необходимости зеркало в renderer).
- Уровни: `debug`, `info`, `warn`, `error`, `fatal` (названия могут быть согласованы с выбранной библиотекой). По умолчанию — `information`. Режим Verbose включается в настройках и пишет расширенно stdout/stderr дочерних процессов.
- Стратегия файлов (гибридная):
  - Имя файла: `FluxAlloy-YYYYMMDD-SESSIONID.json` или `.log` с JSON (SESSIONID — уникальный идентификатор запуска, например, timestamp).
  - Если файл сессии превышает 10 МБ, создаётся следующий с суффиксом `-2`, `-3` и т.д.
  - Хранятся последние 100 сессий; удаление сессий старше 7 дней.
- Консольный вывод разработческой сборки — цветной по уровню (политику согласовать с выбранной библиотекой).
- **Crash / необработанные исключения:** при падении главного процесса или необработанной ошибке в renderer — сохранять **crash report** минимум: stack, версия приложения, ОС, время; при желании использовать **electron crashReporter** или **Sentry‑подобный** сборщик на этапе продукта. Нативные **минидампы `.dmp`** — **опционально** на Windows, не как блокер для macOS/Linux.
- **Диалог ошибки с поиском**: При любой ошибке ffmpeg/yt-dlp (ненулевой код возврата или критическая строка в логе) показывается диалог с текстом ошибки, кнопками «Искать в Google» (открывает `gemini.google.com`), «Искать в DeepSeek» (открывает `chat.deepseek.com`) и «Закрыть». При нажатии на любую кнопку поиска текст ошибки автоматически копируется в буфер обмена.
- Экспорт для поддержки: кнопка в диалоге «О программе» упаковывает логи текущей и предыдущей сессий, **crash report(s)** если есть, версию программы, информацию об ОС в ZIP‑архив `FluxAlloy-Support-YYYYMMDD-HHmmss.zip`.
- Кнопка «Открыть папку с логами» в настройках.

Отдельного требования **.NET Desktop Runtime для основного приложения нет**. Утилиты **диагностики** на Electron или небольшом Node-скрипте — по необходимости (исторический C#‑SetupHelper может существовать как вспомогательный только если сохранён в скриптах сборки).
- При первом запуске приложения выполняется сценарий **§3** (движки, хеши) без установки CLR.

### 19.2 macOS

- **`.dmg`** (или **ZIP**‑распаковка приложения **`FluxAlloy.app`**) с подписью и hardened runtime по мере наличия Apple Developer Program; приложение включает **универсальный или парный** билд ffmpeg/yt-dlp под платформы.
- **Не** использовать Mac App Store в канонической схеме ТЗ без отдельного решения продукта.

### 19.3 Linux

- По минимуму: **архив `.tar.xz` / `.AppImage`** (или `.deb`/Flatpak‑контуры на будущее) с приложением и `bin`‑движками под **`linux-x64`**; при расширении матрицы — **`linux-arm64`**.
- После установки/распаковки — тот же сценарий **§3** (пути к двоичным chmod +x там, где нужно).

### 19.4 Общие требования

- В каждой сборке входит **локальная или подкачиваемая** пара **`ffmpeg`** + **`yt-dlp`**, версии платформы в имени ресурсов или конфигурации сборки (**см. §3**).
- Версионирование и блок «О программе» (**§4.5**) — из `package.json` + механизм внутреннего билд‑номера (скрипт CI или generate).

## 20. Пресеты (структура и хранение)

- Пользовательские пресеты — JSON в папке `Presets`.
- Системные пресеты вшиты, не удаляются, можно клонировать для редактирования.
- Импорт/экспорт всей конфигурации — пункты в меню «Сервис» (единый JSON или ZIP).

## 21. Архитектура и качество кода

- **Основной язык:** **TypeScript** (renderer и main). **Строгость:** `eslint` + `typescript` **`strict`** (или постепенная миграция к strict с исключением в редких модулях — фиксируется в репозитории).
- **Разделение:** Main (IPC, процессы, FS, окна), Preload (**contextBridge only**), Renderer (React-компоненты, состояние, вызовы только через описанный API).
- **Модули:** по доменным сервисам (**ytdlp**, **ffmpegRunner**, **ffprobe**, **presets**, **settingsStore**, …). Принципы **SOLID** и тестируемость (Vitest/Jest для чистых модулей, при необходимости e2e Playwright позже).
- Комментарии на русском в публичных API и сложной логике main/renderer.

## 22. Ожидаемый результат

Исполнитель создаёт **полный монорепозиторий FluxAlloy** (или принятая в команде структура каталогов) с **рабочими целями** `npm run dev` и **сборкой артефактов** хотя бы для **Windows**; по мере готовности — **macOS** и **Linux** той же веткой кода. Упаковка соответствует **§19**. Версия в «О программе» из **§4.5** и метаданных сборки.

Проект исторически мог включать **`FluxAlloy.Core` на .NET** — перенос логики в TypeScript/Python и т.д. **не блокируется** каноном, но **пользовательское приложение** — **Electron + React + TS** (**§2**).

Прогресс отмечается в `IMPLEMENTATION_CHECKLIST.md`.
